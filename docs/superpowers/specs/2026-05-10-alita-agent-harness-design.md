# Alita Agent Harness 架构规范

日期：2026-05-10  
状态：Phase 1 已落地，后续能力持续迭代  
适用范围：Alita的 Agent、工具节点、模型调用、权限与执行系统

## 1. 定位

Alita 不是单纯的聊天软件，也不是把多个工具堆在一个界面里。它的核心是让 AI Agent 在受控环境中理解用户目标、规划流程、调用工具、检查结果并把产物保存到工程里。

因此系统需要一个统一的 Agent Harness 层。

```text
Alita
  = Windows 桌面工作台
  + 模型运行层
  + Alita Agent Harness
  + 可插拔工具节点系统
```

其中模型只是推理能力来源，LangGraph 只是流程编排实现之一。真正决定 Agent 是否稳定、可控、安全、可扩展的是 Alita Agent Harness。

## 2. Harness 的职责

Alita Agent Harness 负责模型外部的工程约束和执行环境，至少包含以下职责：

- 意图理解：把用户自然语言请求转成结构化任务目标。
- 输入完整性检查：判断是否缺少文件、参数、输出格式或用户确认。
- 上下文管理：组织聊天记录、工程文件、工具说明、运行历史和节点输出。
- 工具选择：根据工具 manifest 和任务目标选择可用工具。
- 节点规划：生成自上而下的数据流节点图。
- 执行编排：按依赖关系运行节点，处理分支、汇合、取消和重试。
- 权限控制：限制工具访问路径、网络、脚本执行和输出目录。
- 安全审查：对临时脚本、高风险操作和外部工具调用做审查。
- 结果验证：检查节点产物是否存在、是否为空、是否符合预期格式。
- 运行记录：保存每次运行、每个节点、每个 artifact 和失败原因。
- 反馈循环：失败后让 Agent 基于错误信息调整计划或请求用户确认。

Harness 不直接等同于模型，也不直接等同于 UI。它是连接模型、工具、工程文件和用户确认的控制层。

## 3. 与 LangGraph 的关系

LangGraph 可以继续作为后台 Agent 状态机和流程编排框架，但不能成为整个系统的唯一抽象。

LangGraph 负责：

- Agent 状态机。
- 多步骤任务流转。
- 规划、执行、观察、修正这类循环。
- 需要持久化时的状态检查点。

Alita Agent Harness 负责：

- 工具 manifest 协议。
- 工具权限与安全策略。
- 工程文件边界。
- 节点图数据模型。
- 前端事件协议。
- 模型运行后端选择。
- 运行日志和 artifact 管理。
- 用户授权与审计记录。

关键原则：

- 前端不绑定 LangGraph。
- 工具协议不绑定 LangGraph。
- 模型运行层不绑定 LangGraph。
- LangGraph 内部节点只能通过 Harness 暴露的接口访问工具、文件和模型。
- 如果后续替换 LangGraph，工具 manifest、节点图、工程文件和前端 UI 不应大规模重写。

## 4. 分层架构

```text
前端 UI
  聊天区 / 节点画布 / 首选项 / 工程管理
        ↓
Tauri Rust Core
  窗口 / 工程文件 / 首选项 / 本地安全边界 / sidecar 生命周期
        ↓
Alita Agent Harness
  意图、上下文、工具目录、权限、执行、验证、日志、反馈循环
        ↓
模型运行层
  llama.cpp / 外接 API / 后续其他本地推理后端
        ↓
工具运行层
  MarkItDown / 文档工具 / 图片工具 / 音频工具 / 临时脚本节点
```

第一版允许 Harness 的实现分布在 Rust、Python 和前端之间，但职责必须清晰：

- Rust 是本地可信边界，负责应用配置、工程路径、sidecar 启动、模型运行和高风险系统能力。
- Python sidecar 承载 Agent 编排、工具适配器、节点执行和模型客户端。
- 前端只展示状态、收集输入和提交用户授权，不直接执行工具。

## 5. Harness 内部模块

### 5.1 Intent Router

负责把用户输入分类为普通聊天、工程操作、文档处理、工具流程生成、节点执行、设置修改等任务类型。

第一版要求：

- 用户未提供必要文件时，必须追问。
- 用户提供文件和任务目标后，可以生成节点图。
- 普通聊天不应生成工具流程。

### 5.2 Context Manager

负责组织模型可见上下文，避免简单地把所有历史和文件全部塞进 prompt。

上下文来源包括：

- 当前用户消息。
- 最近聊天摘要。
- 当前工程路径和工程元数据。
- 附件元数据。
- 工具 manifest 摘要。
- 节点运行结果。
- artifact 引用。

第一版暂不做动态上下文长度切换，默认本地模型上下文按 16K 规划。

### 5.3 Tool Registry

负责读取并汇总所有工具包 manifest。

每个工具必须声明：

- `tool_id`
- 名称和描述
- 来源和许可证
- 运行时
- 能力标签
- 可调用 operation
- 输入输出 schema
- 权限需求
- 安全策略
- 错误码
- 节点模板

Agent 只能基于 Tool Registry 中存在且已启用的工具生成调用计划。

### 5.4 Tool Invocation Gateway

负责把 Agent 的工具调用请求转成受控执行。

网关必须执行：

- 工具是否存在。
- 工具是否启用。
- operation 是否被 manifest 声明。
- 输入是否符合 schema。
- 文件路径是否在允许范围内。
- 输出是否写入允许 artifact 目录。
- 网络、插件、脚本能力是否被策略允许。
- 超时、错误码和结果格式是否标准化。

任何工具都不应绕过这个网关直接访问工程文件。

### 5.5 Execution Orchestrator

负责执行节点图。

第一版规则：

- 节点按依赖拓扑顺序执行。
- 支持分支和汇合。
- 每个节点开始、完成、失败都要产生事件。
- 支持取消运行。
- 支持重试失败节点。
- 支持从某个节点重新运行下游节点。
- 节点执行结果写入 run journal。

后续版本可以把这部分进一步迁移为 LangGraph 状态机，但外部协议保持不变。

### 5.6 Safety Gate

负责高风险动作前的审查和授权。

第一版必须覆盖：

- sidecar 本机接口令牌鉴权。
- 工具禁用状态拦截。
- run_id 和 node_id 路径安全校验。
- 工具输入路径限制。
- 工具输出目录限制。
- MarkItDown 禁止网络输入和插件。

后续版本必须覆盖：

- 临时脚本节点 AI 自审。
- 用户确认权限。
- 脚本沙箱。
- 网络访问授权。
- 删除、覆盖、批量写入等高风险文件操作。

### 5.7 Result Verifier

负责检查节点是否真的完成了预期产出。

第一版验证规则：

- 文档转换节点输出 Markdown artifact。
- artifact 文件必须存在。
- 结果文本不能为空。
- 失败时必须返回标准错误事件。

后续可以加入：

- 文件格式验证。
- 文档摘要质量检查。
- 模型输出结构校验。
- 自动修复建议。
- 用户确认通过后再继续下游节点。

### 5.8 Observability / Run Journal

负责记录 Agent 和节点执行过程。

必须记录：

- run id
- task id
- 节点 id
- 节点状态
- 开始和结束时间
- artifact 引用
- 错误信息
- 节点输出摘要
- 运行模式：完整运行、失败重试、从节点重跑

这些记录既服务于调试，也服务于用户信任和后续 Agent 自我修正。

## 6. 工具接入规范

无论工具是自研还是来自 GitHub，都必须作为可插拔工具包接入。

工具接入流程：

1. 新建独立 `tool-packages/<tool>/manifest.json`。
2. 明确工具能力、operation、权限和安全策略。
3. 编写工具适配器，不让 Agent 直接调用第三方库。
4. 通过 `ToolInvocation -> ToolResult` 协议执行。
5. 在首选项中显示工具来源、版本、许可证、能力和启用状态。
6. 增加单元测试覆盖成功、失败、权限拒绝和路径边界。
7. 如果工具来自外部开源项目，必须锁定版本并记录上游仓库。

工具之间不直接互相调用。工具组合由节点图和 Execution Orchestrator 完成。

## 7. 临时脚本节点规范

临时脚本节点是 Harness 的高风险能力，不能和固定工具节点使用同一套默认执行权限。

第一版仅保留协议和 UI 安全状态展示，不执行脚本。

后续启用执行前必须满足：

- Agent 生成脚本后先做安全自审。
- 安全自审结果写入节点详情。
- 明确列出需要的权限。
- 用户确认后才能运行。
- 运行必须在受限目录和受限权限下进行。
- 所有输入、输出、命令、错误和产物必须写入 run journal。
- 默认禁止网络、删除、系统目录写入和长时间后台进程。

## 8. 模型运行策略

Harness 不假设只有一种模型后端。

第一版优先：

- 本地 llama.cpp。
- 默认上下文长度暂定 16K。
- 模型文件存储在用户配置的模型目录，不放进软件开发项目目录。

后续扩展：

- 外接 API。
- 其他本地推理后端。
- 多模型角色分工。
- 向量模型和多模态模型。

模型运行层只提供统一接口，不能直接决定工具权限和工程文件访问。

## 9. 当前实现状态

当前已经具备的 Harness 基础：

- Windows Tauri 桌面窗口。
- 工程文件系统和首选项。
- 本地模型目录和 llama.cpp 运行边界。
- Python sidecar。
- LangGraph 基础 Agent 流。
- 节点画布和节点运行事件。
- Run journal。
- MarkItDown 工具包。
- 工具 manifest 扩展。
- Python Tool Registry，可读取工具包 manifest，并包含虚拟文档输入工具。
- Tool Invocation Gateway，会按 manifest 校验工具、operation 和输入 schema。
- Execution Orchestrator 会在运行前预检 fixed_tool 的 toolRef。
- Result Verifier 会检查关键节点输出、artifact 列表和真实文件存在性。
- 标准 Harness 错误模型，失败事件包含 `errorCode` 和可读错误信息。
- 临时脚本节点已预留 `scriptReview` 安全审查状态协议，默认仅可审查、不可执行。
- 前端节点弹窗可以展示最近运行错误码、临时脚本安全审查摘要和所需权限。
- sidecar 本机令牌鉴权。
- 工具禁用状态拦截。
- 路径安全校验。

仍需规划的 Harness 能力：

- 工具执行超时的强制中断。
- Agent 失败后自动修正策略。
- 临时脚本节点的 AI 自审、用户授权和受限沙箱执行。
- 更完整的运行详情 UI，包括 run journal 浏览、artifact 预览和失败重试建议。
- 工具依赖隔离和版本管理。

## 10. 验收标准

后续所有 Agent 或工具相关功能，如果要认为符合 Alita Agent Harness 架构，至少应满足：

- Agent 不能绕过 Harness 直接访问工具。
- 工具不能绕过 manifest 暴露能力。
- 节点执行必须产生可追踪事件。
- 工具输出必须进入工程 artifact 或明确的安全目录。
- 高风险动作必须有权限策略。
- 用户关闭或禁用的工具不能被执行。
- 失败必须有标准错误事件和可读原因。
- 前端只展示和确认，不直接执行本地工具。
- LangGraph 内部变更不能破坏前端节点图协议。

## 11. 下一阶段建议

下一阶段建议优先做三件事：

1. 让 Agent 规划流程时读取结构化 Tool Registry 能力，而不是依赖硬编码节点模板。
2. 设计临时脚本节点的 AI 自审、用户授权和沙箱执行流程，但默认仍保持不可执行。
3. 增强失败后的自动修正策略，让 Agent 能基于标准错误码调整计划或请求用户确认。

这三项能把当前系统从“受控执行节点流程”推进到“可自我修正、可授权扩展、可审计的 Agent Harness”。
