# Agent Kernel Refactor Design

## Purpose

This document evaluates the proposed Alita Agent algorithm refactor against the current `codex/agent-algorithm-refactor` branch and turns it into an implementable design.

The external proposal is directionally correct: Alita should move from rule routing plus fixed document templates toward an Agent Kernel built around `GoalSpec -> Task DAG -> Tool Binding -> Execution Kernel -> Verifier -> Replanner -> Memory/Skill`.

The proposal is not fully accurate as a description of the current runtime code. Several modules it discusses, including `intent.py`, `task_planner.py`, `privacy.py`, `web_search.py`, `plan_feedback.py`, script approval endpoints, and research execution, exist in prior design/plan documents but are not present in the active Python runtime. The implementation plan must therefore start from the actual current code, not from those planned-but-unimplemented modules.

## Current Runtime Reality

### Implemented Runtime Modules

The current Agent runtime is concentrated in these files:

- `python/agent_service/graph.py`: top-level LangGraph route with `chat`, `missing_input`, and `document_task`.
- `python/agent_service/execution.py`: sequential graph runner with topological sorting, run cancellation, run journal writing, failed-only rerun, from-node rerun, and document-flow execution.
- `python/agent_service/tool_registry.py`: manifest loader plus a virtual `document.receive_attachment` tool.
- `python/agent_service/tool_execution.py`: hardcoded executor for `document.markitdown_convert` and `document.typst_compile`.
- `python/agent_service/result_verifier.py`: fixed node-id verifier for non-empty outputs and artifact existence.
- `python/agent_service/model_client.py`: llama.cpp-compatible chat and streaming client.
- `python/agent_service/app.py`: FastAPI routes for health, chat, graph execution, graph cancellation, and ASR.
- `python/agent_service/asr.py`: Qwen ASR status and transcription service wrapper.

The frontend runtime is concentrated in these files:

- `src/app/App.tsx`: application-level orchestration for projects, messages, graph runs, artifacts, preferences, and voice input.
- `src/app/backendEvents.ts`: reducer for backend event streams.
- `src/features/task/useTaskEvents.ts`: direct HTTP/SSE bridge to the Python sidecar.
- `src/features/canvas/NodeCanvas.tsx`: visual graph renderer and graph run controls.
- `src/features/canvas/NodePopover.tsx`: node detail display.
- `src/features/artifacts/*`: artifact preview and open/reveal APIs.

The Tauri layer is concentrated in:

- `src-tauri/src/lib.rs`: app setup, sidecar startup, llama runtime startup, command registration.
- `src-tauri/src/commands.rs`: Tauri commands for projects, preferences, artifacts, attachments, ASR, and message submission.
- `src-tauri/src/project.rs`: `.alita` project persistence.
- `src-tauri/src/preferences.rs`: model library, model assignments, recent projects, tool enablement.
- `src-tauri/src/llama_runtime.rs`: local llama.cpp startup and sidecar model environment.
- `src-tauri/src/sidecar.rs`: packaged sidecar startup and auth token propagation.

### Planned But Not Implemented Runtime

The repository contains designs and plans for richer routing and research behavior, especially:

- `docs/superpowers/specs/2026-05-18-agent-intent-and-web-research-design.md`
- `docs/superpowers/specs/2026-05-18-task-planner-design.md`
- `docs/superpowers/plans/2026-05-18-agent-routing-task-planner-implementation-plan.md`

However, the corresponding Python runtime files are absent:

- `python/agent_service/intent.py`
- `python/agent_service/task_planner.py`
- `python/agent_service/privacy.py`
- `python/agent_service/web_search.py`
- `python/agent_service/plan_feedback.py`

The external proposal should be read as a useful architecture review plus future-state direction, not as an exact inventory of currently shipped code.

## Technical Assessment Of The Proposal

### Correct Findings

The proposal correctly identifies the central weakness: Alita has a strong product and execution scaffold, but the Agent algorithm core is still simple and narrow.

These findings match the current code:

- Requirement understanding is keyword/rule based in `graph.py`.
- The document workflow graph is fixed and generated by `_create_document_graph()`.
- The LangGraph usage is primarily a router, not a durable multi-step reasoning state machine.
- `ToolManifestSpec` is richer than the current executor uses.
- `ToolExecutor` is hardcoded to two tool IDs.
- Model nodes are not represented by a unified model runtime contract.
- `ResultVerifier` is shallow and tied to fixed node IDs.
- Execution stops on first failure and does not perform local repair or replanning.
- The frontend application state is concentrated heavily in `App.tsx`.
- Python, TypeScript, and Rust schemas are manually synchronized.

### Findings That Need Correction

These claims do not match the current active runtime:

- `intent.py` is not implemented.
- `task_planner.py` is not implemented.
- Web research runtime modules are not implemented.
- `ResearchFlowExecutor` is not implemented.
- `PlannedTaskExecutor` is not implemented.
- Script approval routes such as `/agent/scripts/approve` and `/agent/scripts/reject` are not implemented.
- `ScriptReviewState` exists only in a minimal form: `status`, `summary`, and `permissions`.
- The active `NodeType` union does not include `planning` or `temporary_script`; it includes `temporary_placeholder`.

These corrections matter because starting implementation from the external proposal as-is would create conflicts with the actual code and duplicate prior planning work.

### Recommendations To Keep

The following recommendations should be adopted:

- Introduce `GoalSpec` before planning.
- Keep the current document workflow stable while adding the new kernel path incrementally.
- Separate internal `TaskGraph` from UI-facing `NodeGraph`.
- Add manifest-driven tool binding and execution.
- Treat model calls as a first-class runtime similar to tools.
- Expand verification from fixed node-id checks into schema, artifact, semantic, and final checks.
- Add a local replanner after the first new execution kernel is stable.
- Make network access an explicit permission category before implementing web research.
- Add Agent eval tests before expanding planner complexity.

### Recommendations To Defer

The following recommendations are valuable but should not be in the first implementation slice:

- Temporary script sandbox execution.
- Full Web research workflow.
- MCP runtime support.
- Memory and skill library.
- Frontend hook decomposition, except for minimal changes required by new events.
- Schema code generation across Python, TypeScript, and Rust.
- Parallel node scheduling.

These are not rejected; they are deferred because the first slice must establish stable contracts and preserve the working document flow.

## Target Architecture

The long-term Agent Kernel should use this pipeline:

```text
User Message
  -> GoalSpec Parser
  -> Context Manager
  -> Planner V2
  -> Tool Resolver / Model Resolver
  -> TaskGraph Validator
  -> Graph Compiler
  -> Execution Kernel
  -> Node Verifier
  -> Final Verifier
  -> Replanner
  -> Memory Writer / Skill Library
```

The main design rule is that internal task planning and UI graph rendering must not be the same model.

`TaskGraph` is the internal execution contract. It contains objectives, input/output schemas, tool bindings, model bindings, success criteria, risk levels, and retry policy.

`NodeGraph` is the UI projection. It contains node positions, display names, ports, statuses, artifact references, and recent run records.

## Core Data Contracts

### GoalSpec

`GoalSpec` is the structured interpretation of the user's request.

Fields:

- `goal`: concise user-facing goal.
- `task_type`: one of `chat`, `document_processing`, `research`, `local_file`, `content_creation`, `code_task`, `automation`, `unknown`.
- `deliverable`: expected output, such as `chat_answer`, `markdown_report`, `pdf_report`, `artifact_set`.
- `constraints`: language, style, length, audience, output format, or other requirements.
- `success_criteria`: concrete checks the final result must satisfy.
- `required_context`: context categories needed by planning.
- `missing_inputs`: required inputs that are not available.
- `risk_level`: one of `read_only`, `local_write`, `local_modify`, `destructive`, `network`, `external_comm`, `system`.
- `permissions_required`: permission names needed by the task.
- `needs_web`: whether external network access is required.
- `needs_user_confirmation`: whether planning or execution must pause for user confirmation.
- `confidence`: parser confidence between `0.0` and `1.0`.

First implementation should support deterministic parsing for:

- ordinary chat
- document task with attachments
- document task missing attachments
- local report generation
- explicit PDF/Markdown deliverables
- obvious network requests marked as blocked or requiring confirmation, not executed

### ContextBundle

`ContextBundle` is the planner input assembled from project and runtime state.

Fields:

- `project_path`
- `project_summary`
- `attachments`
- `available_tools`
- `disabled_tool_ids`
- `model_assignments`
- `previous_runs`
- `relevant_artifacts`
- `constraints`

The first version should avoid broad context retrieval. It should only include:

- the current message
- selected attachments
- tool registry summaries
- project path and artifact directory

Important current-code constraint: `/agent/message` and `/agent/message/stream` do not receive `disabled_tool_ids`; the frontend only sends disabled tool IDs when running an existing graph through `/agent/graph/run/stream`. Phase 1 planning must therefore not claim to reject disabled tools during message-to-graph planning unless the frontend payload is expanded. Disabled tool enforcement remains a run-time `RunGraphRequest` concern in Phase 1.

### TaskGraph

`TaskGraph` is the internal DAG.

Fields:

- `graph_id`
- `goal_spec`
- `nodes`
- `edges`

Each `TaskNode` contains:

- `node_id`
- `kind`: `input`, `tool`, `model`, `output`, or `planning`.
- `objective`
- `inputs`
- `outputs`
- `dependencies`
- `success_criteria`
- `risk_level`
- `permissions_required`
- `tool_binding`
- `model_binding`
- `retry_policy`
- `ui`

The first implementation should compile the existing document flow into `TaskGraph` and then into the existing UI `NodeGraph`. It should not try to dynamically plan arbitrary workflows yet.

### ToolBinding

`ToolBinding` is the concrete binding between a task node and a manifest operation.

Fields:

- `tool_id`
- `operation`
- `arguments_template`
- `binding_reason`
- `required_permissions`

The first implementation should bind:

- `document.markitdown_convert` with `convert_local_file`
- `document.typst_compile` with `compile_report_pdf`
- virtual `document.receive_attachment`

### ModelBinding

`ModelBinding` turns model nodes into explicit runtime contracts.

Fields:

- `model_ref`
- `runtime`
- `prompt_template`
- `input_schema`
- `output_schema`
- `temperature`
- `max_tokens`
- `binding_reason`

The first implementation should support:

- `local.content_organizer`
- `local.report_writer`

Both can call the existing `LlamaCppModelClient`, but they should be routed through a common model runtime adapter.

## Implementation Phases

### Phase 0: Baseline And Safety

Goal: establish reliable baseline and avoid implementing on a confused branch.

Work:

- Record current dirty worktree status.
- Do not modify unrelated artifact preview changes.
- Run focused Python tests before changing Agent runtime.
- Add assessment/design documentation.
- Create a new implementation plan after this design is accepted.

Exit criteria:

- The current branch has an explicit design document.
- The first implementation slice is agreed.
- No unrelated worktree changes are reverted.

### Phase 1A: GoalSpec And Internal TaskGraph Foundation

Goal: add kernel contracts while preserving current document flow behavior and without changing the frontend-facing `NodeGraph` schema.

Create:

- `python/agent_service/goal_spec.py`
- `python/agent_service/context_manager.py`
- `python/agent_service/task_graph.py`
- `python/agent_service/graph_compiler.py`

Modify:

- `python/agent_service/graph.py`
- `python/tests/test_graph.py`
- new focused tests for GoalSpec, ContextBundle, TaskGraph, and GraphCompiler

Behavior:

- `/agent/message` and `/agent/message/stream` parse a `GoalSpec`.
- Missing document attachments still emit `input.required`.
- Document tasks still create a graph compatible with the current frontend.
- The graph is produced by compiling a first-version `TaskGraph`.
- Existing document execution still works because UI `NodeGraph` output remains the same.

Non-goals:

- Tool execution refactor.
- Model runtime refactor.
- Verifier replacement.
- Web search.
- Script execution.
- Memory.
- Parallel execution.
- Automatic replan.

### Phase 1B: Tool Resolver And Manifest-Informed Tool Execution

Goal: make the existing document tool execution path extensible without allowing arbitrary manifest entrypoint execution.

Create:

- `python/agent_service/tool_resolver.py`

Modify:

- `python/agent_service/tool_execution.py`
- `python/tests/test_tool_execution.py`

Behavior:

- Tool selection uses manifest operations and capabilities for MarkItDown and Typst.
- `ToolExecutor` dispatches through an explicit adapter map instead of inline `if invocation.tool_id == ...` checks.
- Unknown tools and unsupported operations keep returning the current `HarnessError` codes.
- Existing MarkItDown and Typst tests continue to pass.

Non-goals:

- Dynamic import of arbitrary `entrypoint` modules.
- External CLI generic runtime.
- MCP runtime.
- Temporary script execution.

### Phase 1C: Schema And Artifact Verifier V2

Goal: introduce a verifier that checks declared output keys and artifacts while preserving current failure behavior.

Create:

- `python/agent_service/verifier_v2.py`

Modify:

- `python/agent_service/result_verifier.py`
- `python/agent_service/execution.py`
- `python/tests/test_result_verifier.py`
- `python/tests/test_execution.py`

Behavior:

- Existing `ResultVerifier` behavior remains compatible.
- New verifier logic is data-driven for required values.
- Artifact checks remain strict and return `missing_artifact`.
- Empty required values still return `empty_node_output`.
- The document flow continues to fail fast on verifier errors.

Non-goals:

- LLM semantic verification.
- FinalVerifier.
- Replanner integration.

### Phase 2: Planner V2, Model Runtime, And Deeper Verification

Goal: make task planning more dynamic without abandoning deterministic templates.

Create:

- `python/agent_service/model_runtime.py`
- `python/agent_service/planner_v2.py`
- `python/agent_service/plan_validator.py`
- `python/agent_service/prompt_templates.py`

Modify:

- `python/agent_service/graph.py`
- `python/agent_service/execution.py`
- `python/agent_service/model_runtime.py`
- `python/agent_service/verifier_v2.py`

Behavior:

- Template planner handles known task types.
- LLM planner can be introduced behind a disabled-by-default flag.
- Plan validator checks cycles, dependencies, missing inputs, unavailable tools, and permission mismatches.
- Final verifier checks deliverable existence and basic semantic criteria.

### Phase 3: Replan And Permission Gates

Goal: make failures recoverable and permissions explicit.

Create:

- `python/agent_service/replan.py`
- `python/agent_service/permission_gate.py`
- `python/agent_service/graph_patch.py`

Behavior:

- Common `HarnessError` values map to local patch proposals.
- Disabled tools return actionable messages.
- Missing artifacts can trigger rerun of export nodes.
- Network and destructive permissions are blocked until explicit UI support exists.

### Phase 4: Network Research, Scripts, Memory, And Evals

Goal: add advanced Agent capabilities only after the kernel is stable.

Create:

- `python/agent_service/privacy.py`
- `python/agent_service/web_search.py`
- `python/agent_service/script_sandbox.py`
- `python/agent_service/memory.py`
- `python/agent_service/skill_library.py`
- `python/agent_service/evals/`

Behavior:

- Network tools require explicit permission.
- Privacy guard is mandatory before external requests.
- Script execution requires sandboxing, approval, output capture, and run journal records.
- Memory writes are filtered and project scoped.
- Eval suites cover planner, tool selection, graph execution, verifier, and replan behavior.

## First Implementation Slice

The first slice should be intentionally small:

1. Phase 1A: add `GoalSpec`, `ContextBundle`, `TaskGraph`, and `GraphCompiler`, then compile the existing document graph through the new internal graph.
2. Phase 1B: add `ToolResolver` and move MarkItDown/Typst execution behind explicit operation adapters.
3. Phase 1C: add data-driven schema/artifact verification for the document path.

This slice is valuable because it changes the Agent architecture without changing the user-visible behavior. It creates the kernel seam needed for later planner and replan work.

## Testing Strategy

### Python Tests

Add or extend tests for:

- `GoalSpec` parsing:
  - chat
  - document task with attachment
  - document task without attachment
  - PDF deliverable request
  - explicit network request marked as requiring permission
- `ContextBundle`:
  - includes attachments
  - includes available tool summaries
  - does not read file contents in the first version
  - does not rely on disabled tools during message planning because they are only present in run requests today
- `TaskGraph`:
  - validates dependencies
  - rejects cycles
  - preserves node objectives and bindings
- `GraphCompiler`:
  - compiles document `TaskGraph` into the existing `NodeGraph` shape
  - preserves existing node IDs where possible
- `ToolResolver`:
  - selects MarkItDown for document conversion
  - refuses disabled tools
  - reports missing tools without execution
- `ToolExecutor`:
  - validates operation before dispatch
  - keeps MarkItDown behavior
  - keeps Typst behavior
- `VerifierV2`:
  - validates required output keys
  - validates artifact existence
  - reports structured error codes

### Frontend Tests

Phase 1 should avoid frontend changes unless event payloads change. If frontend changes become necessary, add focused tests for:

- backend event reduction
- graph rendering compatibility
- artifact preview unaffected by new graph metadata

### Rust Tests

Phase 1 should avoid Rust schema changes unless the frontend-facing `NodeGraph` changes. If Rust fields are added, update:

- `src-tauri/tests/domain_tests.rs`
- project save/open tests if persisted schema changes

### Integration Tests

Keep the current document flow integration:

- message with document attachment creates graph
- graph run produces node events
- MarkItDown output is recorded
- Typst output is recorded when Typst is available or mocked
- failed-only rerun still reads prior journal outputs

## Security Direction

The proposal is right that sidecar auth and CORS should become stricter before broader network or script capabilities are added.

Current behavior:

- Python sidecar only enforces `X-Alita-Sidecar-Token` if `ALITA_SIDECAR_TOKEN` is set.
- Tauri packaged sidecar startup sets a token.
- Development sidecar can run without a token.
- CORS currently allows all origins.

Recommended path:

1. Keep development flexibility for the immediate kernel refactor.
2. Add a release-mode or strict-mode testable configuration.
3. Before implementing network research or script execution, require token enforcement and narrow CORS for desktop-origin calls.

Do not bundle sidecar hardening into Phase 1 unless the tests show it can be done without disrupting development flow.

## Frontend Direction

The frontend should eventually split `App.tsx` into focused hooks:

- `useProjectSession`
- `useAgentMessages`
- `useGraphRun`
- `useArtifacts`
- `useVoiceInput`
- `usePreferences`

This is not part of Phase 1. The kernel refactor should first preserve the existing event contract and UI graph shape. After the Agent event surface grows, frontend decomposition becomes necessary to avoid making `App.tsx` the bottleneck for every Agent feature.

## Schema Direction

Long term, the project should adopt a schema source of truth for:

- `AgentEvent`
- `GraphNode`
- `RunGraph`
- `RunGraphRequest`
- `ToolManifest`
- `RunRecord`
- `GoalSpec`
- `TaskGraph`

The likely path is Python Pydantic models exporting JSON Schema, then generating TypeScript types from JSON Schema. Rust can follow later if needed. This is deferred because the first priority is stabilizing the internal Agent kernel.

## Risks

### Risk: Overbuilding The Kernel

The architecture can become too abstract before it proves value. Phase 1 must compile and run the existing document flow only.

Mitigation:

- Keep behavior-preserving tests.
- Do not add LLM planner in Phase 1.
- Do not add memory, web, or scripts in Phase 1.

### Risk: Breaking Current Document Flow

The document flow is the main working Agent loop.

Mitigation:

- Add tests around current graph node IDs and artifacts before refactor.
- Preserve current `NodeGraph` and event shape.
- Refactor behind internal adapters.

### Risk: Tool Manifest Runtime Becomes Unsafe

Manifest-driven execution can accidentally become arbitrary code execution.

Mitigation:

- Phase 1 should use an explicit operation adapter map.
- Manifest metadata selects an adapter, but arbitrary entrypoint import should wait until security policy is implemented.
- Allowed roots remain enforced.

### Risk: External Proposal Scope Is Too Broad

The proposal covers planner, verifier, replan, permissions, scripts, web, memory, MCP, evals, frontend refactor, and schema generation.

Mitigation:

- Treat it as roadmap, not one implementation plan.
- Implement Phase 1 first.
- Write a separate implementation plan for each phase.

## Decision

Adopt the external proposal's architecture direction, but correct its current-code assumptions and implement it incrementally.

The immediate next step should be a Phase 1 implementation plan split into 1A, 1B, and 1C:

- 1A: `GoalSpec`, `ContextBundle`, internal `TaskGraph`, and graph compiler preserving current UI graph.
- 1B: initial tool resolver and manifest-informed document tool execution through explicit adapters.
- 1C: schema/artifact verification for the current document flow.

Do not start with web research, temporary scripts, memory, MCP, frontend decomposition, or schema generation.
